Cloud notebooks reuse the document shell, rail, cells, and output language from
desktop. The host adds the cloud concerns: who is here, whether the document is
connected, who can share, which mode I am in, and which account is active.

This mockup keeps those concerns out of runtime state. The app row carries the
document title plus room, sync, access, mode, and account context; the notebook
row stays the shared `NotebookCommandToolbar` for cell insertion,
package/runtime language, and execution controls.

The workstation mock treats JupyterHub and Outerbounds as providers for the same
nteract workstation object. A workstation registers first as a host-managed peer,
then attaches to the room as a `runtime_peer` only when selected for execution.

## Implementation intent

- Cloud should own an app-level toolbar for presence, document sync, sharing,
  view/edit state, and auth.
- The left side of that app toolbar should read as the document, not as
  collaborator state. Presence belongs with the other action-side room controls.
- The app toolbar should compact before status labels start truncating. The
  document title yields first; at constrained widths the room controls move to a
  second row; connection problems should move to the notice slot when action is
  needed.
- The notebook toolbar should remain the normal notebook-level surface for cell
  insertion, package/runtime language, and execution controls.
- The app row should compose shared primitives where possible:
  `NotebookEditModeButton` and account identity surfaces. Cloud-specific
  presence, transport, and sharing controls can remain host-owned slots.
- Online/offline state should be projected from cloud transport and room state,
  not from `RuntimeStateDoc`.
- Runtime state should describe language and execution capability only; cloud
  transport should not be folded into the Python badge.
- Workstation state is still provisional in app chrome. This mock keeps target
  selection in the workstation rail/detail surface until the host model is ready.
  Provider-specific discovery can later appear in the shared Content rail
  without making the rail decide runtime attachment.
- Current-Python workstations should show Python execution without a managed
  environment badge or package mutation promise.
- Presence should use the quiet avatar stack from the cloud viewer and sit near
  sharing. Visible labels should not repeat room mode; accessible copy can name
  the room population.
- View/edit is a local interaction mode over host permissions. Users with edit
  permission can step into read-only viewing without requesting access again;
  denied users stay gated by the host capability projection.
- Sharing should read as document permissions: public link state, people,
  pending invites, and roles. Avoid generic bubbles, stacked badges, or account
  language that makes collaborators look like runtime actors.
- Sharing is a room-host capability. Desktop and cloud shells should both show
  it only when the active `NotebookRoom` advertises sharing and the current
  identity can use authenticated owner access.
- Sharing controls should appear where the role can act. Non-owner editors and
  viewers need their own quiet pathways: owner shares, viewer requests edit,
  expired sessions renew before changing access.
- Auth controls should explain the current account and recovery path. They
  should not duplicate presence, sharing roles, or notebook runtime state.
- Open-notebook auth and connection problems should sit in the notice slot as
  quiet document signal lines. They should not become raised permission cards
  unless the user is being asked to make a durable access decision.
- Account/session panels should use the same quiet ledger language as sharing:
  signed-in identity, document access, session health, and recovery actions
  without raised avatar bubbles.
- Elements scenarios should continue to model desktop and cloud differences
  through capability facts, not forked shell components.

## Surface split

| Surface           | Shared component         | Host-owned input                            |
| ----------------- | ------------------------ | ------------------------------------------- |
| Presence          | cloud avatar stack       | room participant projection                 |
| View/edit mode    | `NotebookEditModeButton` | ACL plus selected interaction mode          |
| Notebook commands | `NotebookCommandToolbar` | runtime, package, and cell action callbacks |
| Workstation       | cloud rail/detail slot   | selected target, provider, attach status    |
| Connection        | cloud slot               | room transport health                       |
| Sharing           | cloud slot               | invite and ACL actions                      |
| Account           | identity slot            | signed-in user and auth attention state     |
