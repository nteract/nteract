Package management is notebook UI. The rail should let people scan declared
packages as a list, then manage the source only when the host grants that
affordance. This page renders the shared package listing without a live daemon,
Automerge document, or environment solve. Package viewing and package
management are separate shell capabilities; hosted viewer, editor, and owner
fixtures can inspect package state while write actions remain disabled.

## Rail intent

- `NotebookDocumentRail` is the notebook home for declared packages. It wraps
  the shared `EnvironmentPackageSummaryPanel` as a list, not manager cards.
- The catalog uses `NotebookPackageSummaryPanel` for package rail review; uv,
  pixi, Deno, and Conda manager controls stay outside this package listing.
- Manager headers remain app-adapter surfaces because they can write project
  files, approve trust, sync packages, and start rebuild decisions.
- The catalog should keep using existing notebook components, not parallel
  package-manager replicas.
- Live package writes, trust re-signing, daemon sync, and environment rebuilds
  stay behind app adapters. The rendered boundary map names those splits so new
  package surfaces can enter through fixtures before runtime wiring.
