The hosted dashboard is app-level product chrome. It should help a signed-in
user find the next notebook to open without becoming another notebook shell or a
verbose metrics dashboard.

This fixture captures the current product read from the design lab: direction
`1` is the best base, and the useful parts of `5`, `8`, and `10` should become
filtering and precedence around that search-first home. The lower-ranked but
useful `2`, `9`, and `11` directions stay visible as constraints for cleanup,
project grouping, and dense inventory legibility.

## Current read

- `/n` should feel like a notebook switcher first.
- Filtering should borrow from the inbox direction without adopting an inbox
  layout wholesale.
- Activity grouping should make smoke-test clutter easier to scan.
- Pinned or favored notebooks should stay above generated testing noise.
- Triage, project grouping, and table density remain useful constraints, but
  they should not become the primary surface.
- Workstation readiness is room/app context. It can summarize a selected default
  target without owning notebook execution controls.
- Share previews should be revision-aware and safe by default. Private
  notebooks should not leak content through metadata; public notebooks can use
  explicit published data.
- The dashboard can use host-owned app chrome while notebook pages continue to
  use the shared document shell, rail, toolbar, and cell/output surfaces.
