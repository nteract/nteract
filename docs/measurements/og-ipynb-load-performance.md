# OG ipynb Load Performance

**Status:** Measurement, 2026-06-29.

This records the browser/Vite/dev-daemon benchmark shape for opening existing
`.ipynb` files through the real notebook UI. The useful CI gate is behavioral:
cells should attach before the notebook reports synced, final cell count should
be correct, and the visible cell timeline should not shrink. Raw millisecond
thresholds remain measurement data until there is a stable baseline across CI
runners.

## E2E Smoke

`apps/notebook/e2e/og-ipynb-load-performance.spec.ts` records the same browser
milestones and gates these invariants:

- first cell attached before `data-notebook-synced="true"`
- final attached cell count equals expected notebook cell count
- cell-count timeline is monotonic

The committed CI fixture is generated by the test to avoid vendoring external
notebooks. Local runs can include the real matplotlib notebooks by setting:

```bash
NTERACT_OG_IPYNB_LOAD_FIXTURE_DIR=/Users/kylekelley/code/src/github.com/matplotlib/mpl-brochure-binder
```

Single-run local E2E with generated plus real notebooks:

| Fixture | First cell | All cells | Synced | Runtime ready | Timeline |
|---|---:|---:|---:|---:|---|
| `generated-rich-output.ipynb` | 1537 ms | 1747 ms | 1899 ms | 1799 ms | `0 -> 3 -> 13` |
| `MatplotlibExample.ipynb` | 2124 ms | 2124 ms | 2722 ms | 1863 ms | `0 -> 13` |
| `_PreProcess.ipynb` | 1495 ms | 1495 ms | 1556 ms | 1219 ms | `0 -> 11` |

The external notebooks can land straight at the final visible count in a single
sampling interval. That is still acceptable for this gate: cells were visible
before synced, final counts were right, and no visible count shrank.
