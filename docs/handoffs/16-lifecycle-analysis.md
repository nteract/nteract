With the daemon still managing the kernel but syncing to a cloud room instead of the local socket, which kernel lifecycle events are missed? This grounds that question in file:line evidence.

## Verdict

The detection-and-record half of kernel lifecycle is **transport-independent by construction** and survives the swap to a cloud room intact. Every kernel-intrinsic event (death, hang, idle, error, execution-done, graceful shutdown) is sourced from the OS child handle, the ZMQ IOPub/heartbeat sockets, or in-process panic hooks, converted to a `LifecycleSignal`, and written to the `RuntimeStateDoc` without ever touching the sync transport (`daemon-kernel-lifecycle` transport_coupling; `agent-lifecycle-handling` transport_coupling). The cloud room just receives those doc deltas as Automerge sync frames.

But the design has **one fatal gap and three sharp edges** that the cloud-room swap exposes, and they converge on a single deadlock: **the only actor authorized to write kernel lifecycle is the runtime_peer, and the new failure mode is precisely the runtime_peer vanishing** (`runtime-state-model` gaps, `policy.rs:403-405`). When the daemon dies, no surviving room participant can correct the doc, and the cloud room has no watchdog to do it either. A viewer sees a phantom-live, idle kernel forever, with in-flight cells spinning indefinitely.

The blunt summary: **what's MISSED is everything that depends on the agent↔sink link itself, plus all reconciliation that today only runs inside the (now-dead) daemon process.**

## Missed or broken after the swap

**1. Runtime self-disconnect — MISSED entirely (the fatal one).** Detected today only as daemon-socket EOF (`runtime_agent.rs:565-568`), with no agent↔sink heartbeat. There is no detector for a dropped cloud WS, and the EOF teardown policy is *inverted* for a cloud transport: a clean WS close surfaces as `None` → break → `kernel.shutdown()` (`runtime_agent.rs:658-665`), so a transient cloud blip would **kill a healthy kernel** — the opposite of the deliberate "kernel stays running, reconnect" policy applied to framing errors (`runtime_agent.rs:513-563`). And `reconnect_with_backoff` (`:744-769`) / `connect_and_handshake` (`:703-735`) are hardwired to `UnixStream::connect` + `Handshake::RuntimeAgent` with no WS analog.

**2. Cloud-side reconciliation on peer drop — MISSED entirely.** When the runtime_peer WS drops, the room *does* learn it (`webSocketClose`/`webSocketError`, `notebook-room.ts:165-171`) and emits `cloud_peer_left` — but `removePeer` (`notebook-room.ts:635-674`) only deletes the peer, drops its two Automerge sync::State maps via `RoomHostHandle.remove_peer` (`runtimed-wasm/src/lib.rs:523-526`), and broadcasts presence. It **never touches the authoritative `state_doc`.** In-flight executions stay `running` forever; kernel status stays frozen at last-synced value. `RoomHostHandle` exposes *no* execution/kernel mutator (`set_execution_running`/`set_execution_done` exist only on the agent-facing `NotebookHandle`, `lib.rs:401-420`). There is **no `alarm()`, `setTimeout`, `setInterval`, or watchdog anywhere** in the worker. Worse: the `handle_execute_cell` idempotency guard (`lib.rs:807-818`) sees the orphaned exec as still-active and **suppresses re-runs as no-ops**, so a user can't even kick the cell loose.

**3. The policy deadlock — the root cause.** Lifecycle/activity are exclusively runtime_peer-writable (`policy.rs:161-166`). Editor and Owner are blocked by `validate_comm_state_only_runtime_delta`: `if before.state.kernel != after.state.kernel { return Err(... "daemon-owned") }` (`policy.rs:403-405`); Viewer is read-only (`policy.rs:127-138`). When the runtime_peer is the actor that died, **no surviving scope can flip lifecycle to Error/Disconnected.** The doc has no `Disconnected`/`PeerGone`/`Stale` variant and no heartbeat/`last_seen` field (`types.rs:272-286`), so "kernel crashed" and "host closed the laptop lid" are indistinguishable — and a disconnect produces no doc state at all.

**4. Interrupt / restart — PARTIAL (trigger MISSED).** The kernel-side *effect* (idle after interrupt, `Connecting` after restart) survives, but both are kicked by inbound `RuntimeAgentRequest` frames over the daemon socket (`runtime_agent.rs:169-193, 923, 1059`). With no inbound request channel on the cloud room, these user-initiated lifecycle actions never reach the agent.

**5. Best-effort egress, no replay guarantee.** The `state_changed_rx` arm does `break` on writer send error (`runtime_agent.rs:635-643`), tearing down the loop (and kernel) if the cloud writer errors mid-flush. A `KernelDied` delta written while the WS is down has no buffer-and-replay beyond Automerge's own resync — and the resync kick (`state_kick_tx`, `:552`) only fires on the UDS reconnect path. So a death that coincides with a blip may never reach the room.

## Design requirements (ordered)

1. **Add a sink-liveness heartbeat on the agent↔room link, and split the EOF teardown policy by transport.** A cloud WS drop must NOT fall through `runtime_agent.rs:565` into `kernel.shutdown()`. Default policy for a cloud sink: keep the kernel + queue + local doc alive and attempt reconnect (mirror the framing-error branch `:513-563`), never the clean-EOF kill branch. This is the highest-leverage fix because the current default actively destroys healthy kernels. (gap: `agent-lifecycle-handling` 1)

2. **Implement a WS reconnect/re-auth analog of `reconnect_with_backoff` + `connect_and_handshake`** (`runtime_agent.rs:703-769`), and trigger the full-resync kick (`state_kick_tx`, `:552`) on cloud reconnect, not just UDS reconnect. Without this the agent spins redialing a UDS nobody reads, or falls through to teardown. (gaps: `agent-lifecycle-handling` 2, 4)

3. **Add the cloud-room safety net: a DurableObject watchdog that reconciles the authoritative `state_doc` on runtime_peer departure.** This is the only place the net can live, because the daemon's death IS the trigger — daemon-side `set_execution_done(false)` (`runtime_agent.rs`/`kernel_state.rs`) never runs. Concretely:
   - Thread the **peer scope** through `removePeer` → `room-materializer.removePeer` → `RoomHostHandle.remove_peer` (today all take only a `peerId` string, `notebook-room.ts:635-674`, `lib.rs:523-526`), so the room can tell "the runtime owner left" from "a viewer left." (gap: `cloud-room-disconnect` 4)
   - Give `RoomHostHandle` a reconciliation mutator (it has none today) that, on runtime_peer loss, marks running/queued executions failed/abandoned and flips lifecycle to a terminal/disconnected value. (gaps: `cloud-room-disconnect` 1, 2)
   - Use a DO `alarm()` (none exists today) as the timer mechanism, optionally gated on a grace period to tolerate transient reconnects before reconciling. (gap: `cloud-room-disconnect` 3)

4. **Resolve the policy deadlock — pick one and ship it (this gates 3 being legal):**
   - (a) Grant the room host / designated Owner a *narrow* authority to write lifecycle → Error/Disconnected and to terminalize in-flight executions on runtime_peer departure, relaxing `policy.rs:403-405` for exactly that transition; **or**
   - (b) Add a model-level liveness field — a `Disconnected`/`PeerGone` `RuntimeLifecycle` variant and/or a `heartbeat`/`last_seen` timestamp on `KernelState` (`types.rs:272-286`) — that the room is authorized to stamp.
   
   Recommendation: do both — (b) for the model so viewers can distinguish gone-but-recoverable from dead, and (a) scoped to the room host so the watchdog in 3 has a legal writer. Without one of these, 3 is structurally forbidden by `policy.rs:403-405`. (gaps: `runtime-state-model` 1, 5)

5. **Add the inbound request channel on the cloud room** (the Tier-2 hosted REQUEST dispatch, referenced as a separate work item) so interrupt/restart/launch `RuntimeAgentRequest`s reach the agent. Detection of the kernel-side effect already survives; only the trigger is missing. (gaps: `daemon-kernel-lifecycle` 3, `agent-lifecycle-handling` 5)

6. **Guarantee terminal-delta delivery across a blip.** Don't `break` the loop (and kill the kernel) on a single writer send error (`runtime_agent.rs:635-643`); buffer the change and replay on reconnect. Confirm the cloud room reliably resyncs a `KernelDied`/Error delta written while disconnected, rather than relying on the daemon-specific `coordinator_sync_state` + reconnect kick (`:102`). (gap: `daemon-kernel-lifecycle` 2)

7. **Fix the reconnect-recovery contract for a *new* runtime_peer.** When a fresh daemon attaches to a room with stale `running` executions left by a dead peer, the system today relies on it *choosing* to write terminal status rather than guaranteeing it — and the idempotency guard (`lib.rs:807-818`) blocks re-runs in the meantime. Either have the watchdog (3) clear stale state before/at re-attach, or require the joining peer to reconcile as part of handshake. (gaps: `cloud-room-disconnect` 5, 7)

8. **Ensure launch-failure → doc mapping isn't stranded on the daemon.** The launch-time death race and early-exit detection return `Err` from `launch()`, which `launch_kernel.rs` maps to `set_lifecycle_with_error` (`jupyter_kernel.rs:2330-2342`, `launch_kernel.rs:1504-1727`). If launch is driven remotely, confirm that error→doc mapping runs on the kernel-host side. Note the `runt-cloud-peer` prototype only ever writes lifecycle *forward* to `Running(Idle)` on launch (`main.rs:497`) and has **no path that writes Error/Shutdown on kernel death** — `kernel_host.rs` never polls child exit. That prototype needs the same death-detection wiring the daemon already has. (gaps: `daemon-kernel-lifecycle` 4, `runtime-state-model` 4)

Priority ordering for the critical path: **3 + 4 together are the safety net** (and the one failure mode the daemon structurally cannot cover); **1 + 2 prevent the swap from killing healthy kernels**; 5–8 close the remaining trigger/delivery/recovery gaps.

## Area maps

### daemon-kernel-lifecycle

Kernel lifecycle detection lives entirely on the kernel side of the boundary, not the sync-sink side. Every lifecycle event is sourced from one of three intrinsic-to-the-kernel signals: (1) the OS process handle (`tokio::process::Child::wait()` in the process-watcher task, plus PID liveness probes via `kill(pid, None)`), (2) the ZMQ IOPub stream (`iopub.read()` carrying `Status{execution_state}` and error outputs), and (3) the ZMQ heartbeat socket (periodic `single_heartbeat()`). Each detector converts its event into either a direct `RuntimeStateDoc` write (`set_lifecycle`/`set_activity`) or an in-process `LifecycleSignal` enum sent over an unbounded tokio mpsc to the runtime-agent select loop, which then writes the doc via `ctx.state.with_doc(...)`. Crucially, NONE of these detectors touch the daemon socket. The daemon transport (Unix socket / FramedReader / writer) is only used downstream, to ship the already-written RuntimeStateDoc deltas out via Automerge sync (the `state_changed_rx` arm). So the detection-and-record half of the lifecycle is transport-independent by construction: swap the sync sink to a cloud WS room and kernel-death, idle, error, interrupt, and restart all still get detected and written to the doc exactly as today. The one event that IS transport-coupled is "the runtime itself disconnecting": today that's daemon-socket EOF (`framed_reader.recv()` returning None at runtime_agent.rs:566), which breaks the loop and shuts the kernel down. That detection path is specific to the local socket and has no equivalent for a dropped cloud WS connection.

- transport_coupling: Transport-INDEPENDENT (intrinsic to the kernel connection / OS process): all lifecycle DETECTION and all RuntimeStateDoc WRITES. Detection sources are the OS child handle (process.wait at jupyter_kernel.rs:1283, wait_for_pid_exit at 3223), the ZMQ IOPub socket (iopub.read at 1369), the ZMQ heartbeat socket (2582-2589), and the in-process task supervisor panic hooks. Events flow over the in-process `LifecycleSignal` unbounded mpsc (output_prep.rs:459-519) into runtime_agent's select loop, which records them via `ctx.state.with_doc(...)`. The RuntimeStateDoc (RuntimeStateHandle, created at runtime_agent.rs:96-103) is a plain in-memory Automerge doc with no knowledge of where it syncs. None of the detectors or doc writes reference the socket/writer. Transport-COUPLED: only the egress and the inbound-trigger paths. (a) Doc egress: the `state_changed_rx` arm (runtime_agent.rs:616-645) calls generate_sync_message_recovering and send_typed_frame(writer, RuntimeStateSync) over the daemon UnixStream/NamedPipe writer (AgentReader/AgentWriter typedefs at 669-676). (b) Inbound triggers: interrupt, restart/launch, and shutdown are kicked by RuntimeAgentRequest frames read from the daemon socket (framed_reader). (c) Runtime-disconnect detection: framed_reader EOF at line 566 is the daemon socket's close, with no analog for a cloud WS drop.
- gaps:
  - Runtime self-disconnect is detected ONLY as local daemon-socket EOF (runtime_agent.rs:566). A cloud WS room that drops the connection (idle timeout, server eviction, network partition) has no detector here. The agent could keep a live kernel attached to a dead sync sink with no lifecycle write and no teardown, or conversely tear down a healthy kernel on a transient WS blip. There is no agent<->sink heartbeat/keepalive on this link, unlike the kernel<->agent ZMQ heartbeat.
  - KernelDied (and all lifecycle) is recorded in the doc, but its DELIVERY to a cloud room is only as reliable as the WS egress. If the WS connection is down at the moment KernelDied is written, today's code has no buffer-and-replay guarantee for the RuntimeStateSync frame beyond Automerge's own resync-on-reconnect. Whether a cloud room reliably resyncs the death delta after a reconnect needs verification (the local path relies on coordinator_sync_state + state_kick_tx reconnect kick at line 102, which is daemon-reconnect specific).
  - Interrupt and restart/launch lifecycle transitions are triggered by inbound RuntimeAgentRequest frames over the daemon socket. A cloud room must provide an equivalent inbound request channel (the Tier-2 'hosted REQUEST dispatch' work, task 6) or these user-initiated lifecycle events are simply never delivered to the agent. Detection of the kernel-side EFFECT (idle after interrupt, Connecting after restart) survives, but only if the trigger arrives.
  - The launch-time death race (died_rx vs kernel_info_reply at jupyter_kernel.rs:2330-2342) and early-exit detection (try_wait loop) write their result by returning an Err from launch(), which the daemon-side launch_kernel.rs converts into a set_lifecycle_with_error doc write. In a cloud-room world the caller that maps launch() failure to a lifecycle write must still exist and run on the kernel-host side; if launch is driven remotely, confirm the error->doc mapping isn't stranded on the daemon.
  - No explicit 'sync sink unreachable' lifecycle state exists. The doc can record kernel Error/Shutdown but cannot record 'kernel alive, sink unreachable', so a cloud viewer reconnecting cannot distinguish a kernel that died while disconnected from one that is still healthy until fresh IOPub status flows.

### agent-lifecycle-handling

The runtime_agent's lifecycle handling is structurally transport-independent and would survive swapping the sync sink from the local UDS daemon to a cloud WS room. Lifecycle signals (KernelDied / KernelIdle / CellError / ExecutionDone) originate inside the kernel's own in-process tasks in crates/runtimed/src/jupyter_kernel.rs (process watcher line 1295, heartbeat line 2627/2657, IOPub loop exit line 2257, shell loop line 2549, plus panic handlers and the stream/display/output committers' KernelDied sends). They flow over an in-process mpsc UnboundedReceiver<LifecycleSignal> (output_prep.rs:498/509), get selected in the main loop at runtime_agent.rs:575-589, and are dispatched by handle_lifecycle_signal (runtime_agent.rs:1354-1422). Every handler mutates only the in-memory RuntimeStateHandle (Arc<Mutex<RuntimeStateDoc>>) via with_doc and operates directly on the locally-owned `kernel: Option<JupyterKernel>` and `KernelState`. None of this touches the socket. The transport layer is fully decoupled: with_doc compares heads and fires changed_tx (handle.rs:46-50), and a SEPARATE select arm (runtime_agent.rs:617-645) drains state_changed_rx and generates a sync message to whatever writer is installed. The kernel manager (spawn/supervise/restart/interrupt/env pool) and the state-machine writes never reference the connection. So a daemon-managed kernel syncing its RuntimeStateDoc to a cloud room would record kernel death, idle, cell error, interrupt, and restart into the doc identically; the cloud room just receives those doc changes as Automerge sync frames.

The ONE place that is genuinely transport-coupled is the agent's OWN connection dropping (socket read error / EOF), handled at runtime_agent.rs:513-568. On a framing error it calls reconnect_with_backoff (runtime_agent.rs:744-769, 10 attempts, exp backoff) and explicitly KEEPS the kernel + queue + seen_execution_ids + local doc, then kicks a full resync (state_kick_tx.send at line 552). On clean EOF (None, line 565) it breaks the loop and shuts the kernel down (cleanup at 661-663). The critical asymmetry for a cloud-room swap: reconnect_with_backoff dials the UDS path and sends Handshake::RuntimeAgent. It is hardwired to the daemon socket and has no WS analog. If the sink were a cloud WS that dropped, this exact code would either (a) keep trying to reconnect a UDS that the cloud transport doesn't use, or (b) if the WS surfaced as a clean EOF, fall through to line 565 and KILL THE KERNEL on a transient cloud-network blip - the opposite of the deliberate "kernel stays running" policy for socket errors.

- transport_coupling: Transport-INDEPENDENT (operates on RuntimeStateHandle + locally-owned kernel, never the socket): the entire handle_lifecycle_signal dispatch (runtime_agent.rs:1354-1422), all of kernel_state.rs (mark_inflight_executions_failed lives in doc.rs:1342 and is called from the agent; KernelState methods kernel_died/kernel_idle/execution_done/interrupt/clear_queue/process_next all take &mut self + &mut impl KernelConnection), the interrupt/restart/shutdown request handlers' state writes, and the with_doc -> changed_tx -> state_changed_rx notify chain (handle.rs:46-50). The signals are delivered over in-process mpsc channels (output_prep.rs:498) wired from the kernel's own tasks in jupyter_kernel.rs - no transport involved. The state-machine has zero references to the writer or socket. Transport-COUPLED: only the agent's own connection-drop handling. reconnect_with_backoff (runtime_agent.rs:744-769) and connect_and_handshake (:703-735) are hardwired to UnixStream::connect / NamedPipeClient + Handshake::RuntimeAgent; the framed_reader and the outbound sync arm (:617-645, :490-496) write to the AgentWriter half of that socket. The sync send is decoupled by indirection (it writes to `writer`, whatever that is) but the reconnect/handshake and EOF-teardown policy are UDS-specific.
- gaps:
  - Sink-drop teardown policy is inverted for a cloud transport. A clean WS close surfaces as `None` (runtime_agent.rs:565) -> break -> kernel shutdown. A transient cloud-room disconnect would KILL a healthy daemon-managed kernel, the opposite of the deliberate 'kernel stays running, reconnect' policy applied to socket framing errors (:513-563). The Err-vs-None branch encodes UDS-specific assumptions about which disconnects are recoverable.
  - reconnect_with_backoff (runtime_agent.rs:744-769) and connect_and_handshake (:703-735) only know how to redial the UDS / named pipe with Handshake::RuntimeAgent. There is no WS reconnect/re-auth analog, so a dropped cloud room would either spin reconnecting a socket nobody is reading or fall through to teardown. The runtime_peer cloud path is a separate codebase (per task 16) and the agent's recovery loop does not cover it.
  - Lost-signal handling on receiver-closed is silent: KernelDied sends use `let _ =` (jupyter_kernel.rs:1294-1298, 2662) or only warn (iopub :2257). If a cloud-room outage coincides with kernel death and the agent loop has already broken, the KernelDied -> Error doc write may never happen, so the cloud room would never observe the kernel as dead - it would see the last synced state (e.g. Running/Busy) frozen. There is no heartbeat/liveness write into RuntimeStateDoc independent of these one-shot signals.
  - Outbound sync delivery is best-effort against a single writer: the state_changed_rx arm (runtime_agent.rs:635-643) does `break` on a send error, tearing down the loop (and kernel) if the cloud writer errors mid-flush. Lifecycle doc changes generated just before a cloud-room blip can be lost from the wire even though they are durable in the local doc, until/unless a resync round runs - and the resync kick (state_kick_tx, :552) is only triggered on the UDS reconnect path, not on a cloud reconnect.
  - interrupt/restart are request-driven (RuntimeAgentRequest frames). They are marked 'partial' because the state writes are transport-independent but the trigger arrives over the same socket frame channel; a cloud room would need an equivalent inbound request path or these control actions never reach the agent.

### cloud-room-disconnect

The cloud room does NOT reconcile in-flight executions or kernel lifecycle when a runtime_peer connection drops. When any peer socket closes (close, error, failed send, or sync failure), NotebookRoom.removePeer (apps/notebook-cloud/src/notebook-room.ts:635-674) does exactly three things: (1) deletes the peer from the in-memory peers Map, (2) calls materializer.removePeer(peer.id) which forwards to the WASM RoomHostHandle.remove_peer (crates/runtimed-wasm/src/lib.rs:523-526), and (3) broadcasts a cloud_peer_left presence control message. RoomHostHandle.remove_peer ONLY drops the peer's two Automerge sync::State entries (notebook_peer_states, runtime_peer_states). It never touches the authoritative state_doc (RuntimeStateDoc). It does not mark running/queued executions failed or abandoned, does not reset the kernel lifecycle to NotStarted/Error, and does not even know it was the runtime_peer that left (removePeer is passed only a peerId string, never the scope). The RoomHostHandle exposes no execution/kernel-status mutator at all: the only RuntimeStateDoc writes it can author are via receive_runtime_state_sync (replaying the runtime_peer's own synced changes) and handle_execute_cell (creating new queued executions). set_execution_running/set_execution_done exist only on the agent-facing NotebookHandle (lib.rs:401-420), not on RoomHostHandle. So any execution the daemon marked "running" before it died stays "running" in the cloud RuntimeStateDoc forever, and the kernel status the daemon last synced is frozen as-is. There is NO watchdog, heartbeat-liveness check, alarm(), setTimeout/setInterval, or timeout anywhere in the cloud worker (the only "heartbeat" is presence-frame encoding, not liveness). The DurableObject has no alarm() handler. This is precisely the safety net the daemon-side manager cannot provide, because in the runtime_peer-disconnect scenario the daemon may be exactly what died. The daemon's own reconciliation (crates/runtimed/src/runtime_agent.rs:996-1479 and kernel_state.rs calling set_execution_done(eid, false) on KernelDied/interrupt) runs in the daemon process and writes to its local RuntimeStateDoc; if the daemon process is gone, none of it fires, and nothing on the cloud side substitutes.

- transport_coupling: Disconnect DETECTION is fully transport-coupled to the Cloudflare DurableObject WebSocket lifecycle: webSocketClose/webSocketError (hibernation API) and the per-socket close/error listeners (notebook-room.ts:165-171, 213-218) are the only triggers, plus send-failure (trySendFrame at 608-615) and syncPeer-failure (438-466). All are reactions to the WS transport signalling a dropped socket. This part actually works correctly for a cloud room: when the runtime_peer's WS to the DurableObject drops, the room DOES learn the peer is gone and DOES emit cloud_peer_left. What is entirely transport-INDEPENDENT but MISSING is the reconciliation: there is no code anywhere (room host or worker) that, given 'a peer left', reconciles the authoritative RuntimeStateDoc. The room host's remove_peer is intentionally just sync-state cleanup. The execution/kernel reconciliation logic lives only in the daemon process (runtime_agent.rs / kernel_state.rs), reachable only over the daemon's own local path, and it is the daemon's death that triggers the disconnect in the first place. So the cloud side has the disconnect signal but no policy to act on it.
- gaps:
  - Running/queued executions are NEVER marked failed or abandoned when the runtime_peer drops. They stay status 'running' in the cloud RuntimeStateDoc forever, with no watchdog to expire them. Viewers see a perpetually-spinning cell.
  - Kernel lifecycle status is never reset to NotStarted/Error/Disconnected on runtime_peer loss. The last-synced status (idle/busy) is frozen, presenting a phantom-live kernel to all cloud viewers.
  - No liveness mechanism exists: no DurableObject alarm(), no heartbeat-timeout, no idle/stale-execution watchdog. The presence 'heartbeat' is encode-only and is not consumed as a liveness signal server-side.
  - removePeer cannot even tell that the departing peer was the runtime_peer vs a viewer: room-materializer.removePeer takes only a peerId, and RoomHostHandle.remove_peer takes only peer_id. Scope-aware reconciliation (e.g. 'the runtime owner left, fail its in-flight work') is impossible without threading the scope through.
  - The idempotency guard in handle_execute_cell (lib.rs:807-818) actively WORSENS the stuck state: because the orphaned execution is still 'running', a re-run request is suppressed as a no-op, so a user cannot even kick the cell loose by re-running it.
  - This is the one failure mode the daemon-side manager structurally cannot cover: the daemon dying IS the trigger, so daemon-side set_execution_done(false) in runtime_agent.rs/kernel_state.rs never runs. The cloud room is the only place a safety net could live, and today there is none.
  - Reconnect semantics are unspecified for recovery: if a fresh runtime_peer (new daemon) attaches to the same room, nothing forces it to reconcile the stale 'running' executions left by the previous peer; it would need to deliberately write terminal status, and the system relies on that happening rather than guaranteeing it.

### runtime-state-model

The RuntimeStateDoc models runtime liveness entirely through one field: KernelState.lifecycle (a RuntimeLifecycle enum), with KernelActivity carried only inside the Running(activity) variant. "Kernel dead" and "runtime disconnected" are NOT distinct states. Both collapse onto RuntimeLifecycle::Error (generic death) or RuntimeLifecycle::Shutdown (clean stop). There is no NetworkLost / PeerGone / Disconnected variant, no heartbeat timestamp, and no liveness/presence field in the doc. The viewer's only signal that a runtime is gone is reading lifecycle == Error|Shutdown (consumed in notebook-sync/execution_watch.rs:158-162, runtimed-py/session_core.rs:354-371, execution_wait.rs:127-132).

The policy is unambiguous and is the deadlock: lifecycle/activity are runtime_peer-owned. The Editor/Owner (room/host) write path runs validate_comm_state_only_runtime_delta (policy.rs:391), which at policy.rs:403-405 rejects ANY change to state.kernel with the error "scope {editor|owner} cannot write RuntimeStateDoc kernel: daemon-owned". Viewer is read-only (policy.rs:127-138). So the only scope permitted to write kernel lifecycle is RuntimePeer (policy.rs:161-166 comment: "Runtime peers own kernel-facing state: lifecycle/activity, output routing, comm topology"). If the runtime_peer is the process that died/disconnected, no surviving actor in the room is authorized to flip lifecycle to Error/NotStarted. The doc is stuck displaying the last value the dead peer wrote (typically Running(Idle) or Running(Busy)) forever.

Quote, policy.rs:403-405 (Editor/Owner path): `if before.state.kernel != after.state.kernel { return Err(runtime_state_policy_error(scope, "kernel", "daemon-owned")); }`. This is enforced for Editor and Owner alike via the shared validate_comm_state_only_runtime_delta.

Today this is masked because the daemon IS the runtime_peer's manager and detects death out-of-band (process exit -> runtime_agent.rs:1395-1417 writes Error; clean stop -> shutdown_kernel.rs:26 / jupyter_kernel.rs:1415 writes Shutdown). None of that detection lives in the doc or the room; it lives in the local daemon supervising the kernel subprocess over the UDS.

- transport_coupling: Transport-INDEPENDENT (lives in the doc, syncs anywhere): the RuntimeLifecycle/KernelActivity model itself (types.rs), the typed setters (doc.rs set_lifecycle / set_lifecycle_with_error), the resolve_lifecycle read reconciliation, the execution diff/projection (projection.rs), and the write-scope policy (policy.rs). These travel with the CRDT regardless of UDS vs cloud-room WS. The viewer's "kernel gone" derivation (execution_watch.rs) is also transport-independent because it reads only doc lifecycle.

Transport-COUPLED (lives in the local daemon, does NOT travel to a cloud room): every mechanism that actually DETECTS a runtime going away and authorizes the corresponding lifecycle write. (1) Kernel-subprocess death detection is in runtime_agent.rs / jupyter_kernel.rs, bound to the daemon owning the child process; the runt-cloud-peer prototype has no child-exit watcher in kernel_host.rs. (2) Runtime_peer-disconnect detection is the UDS connection-close callback in peer_connection.rs/peer_eviction.rs, and it does not even write lifecycle. (3) The authority to write lifecycle is gated on connecting as RuntimePeer scope (policy.rs:161); a cloud room's editor/owner connection is structurally forbidden from writing it (policy.rs:403-405).
- gaps:
  - DEADLOCK: lifecycle/activity are exclusively runtime_peer-writable (policy.rs:161-166). Editor and Owner scopes are blocked from any kernel write by validate_comm_state_only_runtime_delta (policy.rs:403-405); Viewer is fully read-only (policy.rs:127-138). If the runtime_peer dies or its WS drops, no surviving room participant is authorized to flip lifecycle to Error/NotStarted. The doc is frozen at the dead peer's last value (commonly Running(Idle)), so the viewer shows a live, idle kernel that does not exist and cell executions hang against terminal-reason logic that never fires.
  - No DISCONNECTED state and no liveness signal in the model. RuntimeLifecycle (types.rs:272-286) has no PeerGone/NetworkLost/Stale variant; KernelState has no heartbeat or last_seen timestamp. 'Kernel dead' (Error) and 'runtime disconnected' are indistinguishable, and disconnect produces no doc state at all. A room cannot even tell 'the kernel crashed' from 'the runtime host's laptop closed the lid'.
  - On the local UDS transport, runtime-away is caught by the daemon's connection-close callback (peer_eviction.rs) and by kernel-subprocess supervision (runtime_agent.rs). In a cloud room, the editor/owner peers receive a WS-level peer-gone signal from the room, but (a) the policy forbids them from writing lifecycle, and (b) there is no code that even attempts it. The detection that exists today does not survive the transport swap.
  - The cloud-peer prototype (runt-cloud-peer) only ever writes lifecycle forward to Running(Idle) on launch (main.rs:497) and closes the socket on its own clean exit (main.rs:630). It has NO path that writes Error/Shutdown on kernel death, and kernel_host.rs never polls the child process for exit. A crashed kernel under the cloud peer leaves the doc at Running(Idle) permanently.
  - Recommended resolution requires a policy change plus a writer: either (a) grant the room host / a designated Owner the authority to write lifecycle->Error/NotStarted on runtime_peer departure (relaxing policy.rs:403-405 for a narrow lifecycle-to-terminal transition), or (b) add a liveness field (heartbeat/last_seen or an explicit Disconnected lifecycle) the room can stamp, so the viewer can distinguish gone-but-recoverable from dead. Without one of these, the room is stuck whenever the only authorized lifecycle writer is the actor that vanished.
