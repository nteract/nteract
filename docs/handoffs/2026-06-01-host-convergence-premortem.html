Synthesis - read this first

Most likely failure

The recovery loop never gets closed because "blunt reset already exists" was conflated with "recovery is solved." This is the highest-likelihood failure (verdict-confirmed, daemon-local path masks it in all single-user testing). The reset ACTION exists (useAutomergeNotebook.ts:299-318 daemon:ready bootstrap; reset_sync_state at runtimed-wasm/src/lib.rs:2447). The reset TRIGGER does not. I verified the cloud room ALREADY emits a corrective frame on rejection: notebook-room.ts:553-555 sends `cloud_frame_rejected`, the client receives it at live-sync.ts:311, but presence.ts:48 only handles `cloud_room_ready` and silently drops everything else. So the wire is half-built and inert. Today's reject at runtimed-wasm/src/lib.rs:594-618 returns `Err` on a clone-preview (the real peer_state at :600-606 is never advanced, so the room is undamaged) and the diverged peer re-offers forever via flush_local_changes. The split does not fix this; it relocates and multiplies the rejection surface (every viewer CellsDoc write, every mixed-version legacy-root write). Because the daemon-local path uses per-field actor filtering (runtime_agent.rs:348-352), not message rejection, single-user and desktop testing never reproduce it. The team ships the headline doc split, the re-offer-forever stall survives untouched, and "trivially recoverable" means "recoverable by a mechanism nobody wired."

Most dangerous failure

Silent half-notebook / silent data invisibility from the cells/metadata data-move, in three converging forms that all fail without a crash, log-above-info, or .corrupt rename: (1) In-place reshape LWW-merging the frozen root into garbage — a deadline "copy-and-clear the legacy root" shortcut is invisible to verify-genesis (byte-presence only, xtask/main.rs:1586) and to the canonical seed hash-pin (seed-layer only, notebook-doc/src/lib.rs:2421), and the forward-tolerant reader loads it as-is with no quarantine (lib.rs:1208-1260). (2) Phase-0 soak shipped on faith — a v5 cloud client hits the tolerate branch, reads root `cells` directly via cells_map_id() (lib.rs:2444), finds it empty, renders a blank notebook, and the user types into the legacy root, autosaving an empty-cells snapshot over real CellsDoc content (the #3086 autosave-zeroing class). (3) The .ipynb emitter joining two docs — save_notebook_to_disk reads get_cells() AND get_metadata_snapshot() off one handle (persist.rs:91-102); post-split one subtree reads empty and the zeroing guard (persist.rs:304) only fires on cell_count==0 && load_failed, missing partial-emptiness, so disk and the publish artifact both become half-notebooks, made durable bidirectionally by the file watcher. Most likely costs convergence; most dangerous costs users their work, irreversibly, on the least-likely-to-file-a-precise-bug population.

The hidden assumption

Every failure shares one assumption: that a property proven at the DOCUMENT layer (authorization is all-or-nothing per principal; a rejected write means no good edits to preserve; topology and state can live in separate docs) holds at the LAYERS THAT TOUCH THE DOCUMENT — sync, persistence, lifecycle, recovery, runtime echo-suppression, identity-pointer durability, and scope-currency. The north star is "authorization boundaries equal document boundaries." It silently extends to "EVERYTHING that was incidentally provided by co-location in one document survives being split into many." It does not. Co-location was secretly providing: the comm orphan-state Guard A (doc.rs:2502-2515), the kernel-echo authorship filter's substrate (one doc, distinguishable actor prefixes, runtime_agent.rs:351), closed-comm GC authority (remove_comm deletes topology+state in one op), atomic cells+metadata reads for .ipynb export, and identity-pointer durability via deterministic re-derivation. And the scope that governs each new boundary is itself a connect-time-frozen snapshot (notebook-room.ts:85,199), not a live fact — so "rejected == no good edits" is true only AFTER data lives in the right per-principal doc AND the writing principal's authority is still current. The split makes authorization coarser and more confident at the exact moment it removes the per-message inspection that was the last thing running on every write, while inheriting and amplifying a v1 revocation gap that the document boundary cannot police.

Recommendation

Pursue a MODIFIED, DE-COUPLED split — not the 5-doc all-or-nothing as a single program. The forced-derivation to five documents is logically CONFIRMED (metadata is owner-mutable post-genesis, so genesis-freeze fails and NotebookDoc must split owner-only-metadata vs editor+owner-cells; RuntimeStateDoc's editor comm-state carve-out separately forces CommsDoc). But "confirmed as the logical endpoint" is not "confirmed as one shippable program." Every survived failure that reached HIGH or CRITICAL clusters on TWO things the all-or-nothing framing under-counts: (a) the irreversible cells/metadata data-move (in-place reshape, Phase-0-on-faith empty-notebook, .ipynb half-notebook, mixed-version force-reset, random-pointer fork, the five-seed/dual-validator maintenance ledger), and (b) the recovery half being scoped as a follow-up (re-offer-forever survives, monolithic reset, sever-threshold collision, divergent-genesis reset loop, daemon-local/cloud divergence). The authorization payoff (delete one editor-branch path check, owners already bypass it via can_write_all_notebook_changes) is real but small and DUPLICATED across daemon and WASM validators, while the cost is permanent. The right move: take the two independently-justified, low-risk wins now — CommsDoc (collapses the RuntimeStateDoc carve-out, cheap additive new-doc) and additive owner-only identity pointers — and treat the cells/metadata structural data-move as a SEPARATE, telemetry-gated, reversible-until-proven decision that does not start until the recovery protocol and the export/identity/fleet-floor guards exist. The north star is correct as a direction and wrong as an all-or-nothing delivery: the document boundary cleanly governs WHO-may-write, but it does not subsume version, time, lifecycle, persistence, or scope-currency, and those are exactly where the dangerous failures live.

Failure modes (19), most severe first

CRITICAL ■■■ high likelihood migration

Phase-0 soak shipped on faith: v6 data-move outran the v5 fleet, and silent empty-notebook rendering hid it until users reported lost work

The split landed in the textbook order. Phase 1 added cells_doc_id and comms_doc_id as additive root scalars, registered the new seeds in GENESIS_SEEDS_IN_WASM, and verify-genesis passed. The forward-tolerant reader was already in v5 (lib.rs:1208-1261), so the team checked the box: "v5 tolerates v6, soak is satisfied." Phase 2 dual-wrote. Phase 3 flipped the daemon to author cells into CellsDoc and stopped populating the legacy root cells map. The desktop fleet was fine because desktop ships daemon and frontend as one build (deployment-topology.md:108-111), so every desktop peer crossed to v6 atomically. The bug lived entirely in the cloud topology, which the deployment ADR explicitly designs to mix independently-versioned peers: browser-direct, native-direct, and local-daemon-bridge clients (deployment-topology.md:220-242), each on its own release cadence, none gated on a fleet-wide version floor. What actually happened to a v5 cloud client: it opened a post-move v6 room, hit the tolerate branch (lib.rs:1221, Valid(v) >= SCHEMA_VERSION), and returned the doc as-is with the comment "schema evolution is additive" (lib.rs:1244-1253). But this move was NOT additive, it was the structural reshape the ADR forbids (schema-evolution-and-genesis.md:52-55). The v5 client's cells_map_id() (lib.rs:2444) read the root "cells" map directly, found it empty or absent, and rendered an empty notebook. No crash, no .corrupt, no log line above info level, no error frame. The forward-tolerant reader did exactly what it was built to do, which is the trap: it is engineered to NEVER reject a newer doc, so it cannot distinguish "additive future fields I can safely ignore" from "my entire cells subtree moved to a doc I do not know how to follow." The worst part was the write path. A v5 user, seeing a blank notebook, assumed sync had not loaded yet and started typing. Their edits authored into the legacy root cells map (the only location v5 code knows), which the daemon's post-move policy now rejects, OR worse, autosaved a snapshot whose root cells were empty over a notebook that had real content in CellsDoc. The #3086 class of autosave footgun (schema-evolution-and-genesis.md:132-134) re-ran: a peer with the wrong shape silently overwrote good state. It went unnoticed for weeks because the people who experienced it were the long-tail of un-updated browser tabs and pinned native clients, exactly the population least likely to file a precise bug. It surfaced as "my notebook is empty / I lost my work," not as a stack trace, so it sat in support triage as user error until someone correlated it with the v6 cutover date.

Underlying assumption. That "v5 ships a forward-tolerant reader" is equivalent to "v5 can safely participate in a v6 room." It is not. The tolerate branch only guarantees v5 will not corrupt or reject a v6 doc; it guarantees nothing about v5 reading the right DATA, because tolerance is the absence of rejection, not the presence of pointer-following. The split conflated "old client won't break the doc" (true, additive) with "old client renders the doc correctly" (false, structural). There was no mechanism that observed the actual live fleet and confirmed zero v5 peers before the daemon performed the irreversible data-move; "soak" was a calendar assumption, not a measured invariant.

Early warning signs

  • A spike in info-level logs '[notebook-doc] Loaded forward-version v6 doc ... reading tolerantly' (lib.rs:1227-1231) originating from CLOUD room peers after the Phase-3 cutover. On desktop this line should be near-zero (single-build atomic upgrade); any sustained volume from browser/native clients means live v5 peers are reading post-move docs and seeing empty cells. This log already exists and is the canary, but it is info-level and unmonitored.
  • Cloud rooms showing a non-empty CellsDoc snapshot in R2 while a connected peer's outbound sync frames or autosave snapshot carry an empty or shrinking root cells map. A daemon-side assertion 'a peer is writing root.cells after cutover' or 'snapshot root.cells count dropped while CellsDoc count is positive' would fire on the exact write-path corruption before users notice. Also: support tickets clustering on 'notebook opened empty' / 'lost cells' with timestamps correlating to the v6 deploy.

Design changes

  • Gate the data-move on a MEASURED fleet floor, not a calendar soak. The room host already sees every peer's schema_version per connection. Before the daemon performs the irreversible Phase-3 move for a given room, require an observed invariant: every peer that has connected within the snapshot-retention window reported >= v6. Refuse to author the move into a room that has seen a v5 connection recently. This converts 'soak propagated' from an assumption into a precondition the code enforces per-room.
  • Make the v5 tolerate branch refuse to render structurally-moved docs instead of silently showing empty. Before the v6 cutover ships, release a v5.x point build whose classifier, on encountering Valid(v) > SCHEMA_VERSION AND a cells_doc_id/comms_doc_id pointer present at root, treats the doc as unreadable-by-this-build and surfaces a hard 'please update' state rather than an empty notebook. A present new-doc-id pointer is the unambiguous signal that data moved; tolerance should be conditional on the ABSENCE of known structural-move pointers. This is the real Phase-0 prerequisite, and it must propagate before any move, mirroring invariant 4 (schema-evolution-and-genesis.md:147) but enforced by detection, not trust.
  • Keep the legacy root cells map populated as a read-only mirror through a full deprecation window (dual-write, never dual-stop). The move's irreversibility is what makes silent invisibility catastrophic. If the daemon continues writing cells to BOTH root and CellsDoc for an extended period, a v5 reader sees real (if eventually stale) content, and a v5 writer's root edits can be reconciled rather than lost. Only stop writing the legacy root after the measured fleet floor confirms zero v5 peers. This trades CRDT size for the elimination of the silent-data-invisibility class entirely.
  • Make a v5 client physically unable to write the legacy root after cutover detection. If a v5 peer that has been told (or can detect via the new-doc-id pointer) it is in a v6 room still has only legacy-root write paths, block its writes at the room host with an explicit typed rejection that the v5.x build renders as 'this notebook requires a newer client' rather than accepting the empty-root autosave. This closes the autosave-zeroing footgun even if a stale client slips through.
  • Add a deploy-time and runtime telemetry gate: alert on any forward-version-tolerant load from a cloud peer, and block the Phase-3 deploy flag until that metric is zero across the cloud fleet for the retention window. The log line exists (lib.rs:1227); promote it to a monitored metric tagged by room-host topology so 'are there still v5 peers' is a dashboard number, not a guess.
CRITICAL ■■■ high likelihood migration

Mixed-version cloud room force-resets old-v5 editors and discards their legitimate cell edits during the data-move window

The split shipped in phases exactly as the migration plan prescribed. Phase 0 (forward-tolerant reader) and Phase 1 (additive cells_doc_id / comms_doc_id pointers minted ensure_-style) soaked cleanly for weeks because they are additive and the proven runtime_state_doc_id playbook (notebook-doc/src/lib.rs:1279). The team grew confident and shipped Phase 2/3: the daemon began minting CellsDoc and the new all-or-nothing policy treated CellsDoc as the editor-writable doc and NotebookDoc root as owner-only. The reasoning was the north-star claim: a rejected write means "you wrote a doc you cannot write at all," so blunt reset is lossless because the peer has no good edits to preserve. But cloud rooms are mixed-version by construction (schema-evolution-and-genesis.md:108-111: cloud lets independently-versioned peers share a room). A pinned old-v5 frontend build, the kind that lives in a long-open browser tab, still authors every cell edit into NotebookDoc root cells/{cell_id}/source. It has no code to follow cells_doc_id and write CellsDoc. When that v5 editor typed, its local Automerge applied the splice immediately (local-first, runtimed-wasm/src/lib.rs splice_source path), then offered the change to the room. The room ran the auth preview on a clone (lib.rs:605-606), and because cells now belongs to the owner-only NotebookDoc surface under the new policy, the change failed validation. The exact shape of the failure depends on how the team wired the new policy: either validate_editor_notebook_changes still runs and now its within_cells check (lib.rs:940-947) is inverted relative to where editors are supposed to write, or the editor's can_write bit for NotebookDoc was flipped to false and the has_changes && !can_write gate (lib.rs:594-598) rejects wholesale. Both reject the entire message. Then the recovery design did exactly what it was built to do, and that was the disaster. The room replied RESYNC_REQUIRED with authoritative heads. The v5 client ran the existing blunt-reset block (useAutomergeNotebook.ts:299-318), discarding its local NotebookDoc replica, including the cell text the user had just typed and every other unsynced cell edit batched in that flush cycle, and re-bootstrapped from the room snapshot. The user watched their paragraph vanish. They retyped it. The v5 build authored it into root cells again. Rejected again. RESYNC_REQUIRED again. The per-connection rejection-after-resync counter ticked up, and after N (3) reset loops the room escalated to SESSION_CONTROL Sever and kicked them out entirely, reading a perfectly honest, perfectly authorized editor as malice because reset-then-rediverge is the documented signature of an attacker or a wrong build. The rejected==no-good-edits claim was simply false here: the good edits lived in the same legacy root the policy now rejects, because the data had not finished moving into the per-principal doc on this peer's build.

Underlying assumption. That "a rejected write means the peer has no good edits in that document to preserve" is a property of the all-or-nothing document model. It is not. It is a property of the model AFTER every peer in the room writes the new per-principal doc shape. During the data-move window an old-v5 editor's cell edits are simultaneously legitimate (it has write authority over cells) and rejected (it wrote them into the wrong document, the legacy NotebookDoc root the new policy reads as owner-only). The rejection is caused by version skew, not by lack of authority, so blunt reset destroys real authorized work. The migration plan even named this hazard ("the rejected==no-good-edits claim holds only after data lives in the right per-principal doc"), but the recovery design armed the reset frame against the same rejection class anyway.

Early warning signs

  • Per-connection rejection-after-resync counters climbing and SESSION_CONTROL Sever events firing in cloud rooms, correlated with the client build/schema version of the severed peer: severs cluster on old-v5 frontend builds, not on attackers or new builds. A dashboard of sever-events-by-client-version makes this visible within hours of Phase 2/3 rollout.
  • Notebook auth-preview rejections (the JsError from runtimed-wasm/src/lib.rs:613/618) whose rejected patch path.first() is the cells map specifically, rather than metadata/schema_version/notebook_id. A cells-rooted rejection from an editor scope is definitionally a version-skew rejection, not an authorization violation, because editors are supposed to be allowed to write cells. Logging the rejected root key separates the two classes immediately.
  • User reports of typed cell text reverting or disappearing and a load-flash mid-edit, concentrated among users on stale long-open tabs rather than fresh sessions.

Design changes

  • Gate the recovery/reset frame behind a rejection-class discriminator, not a blanket recoverable-first policy. A rejection whose offending patch is rooted at cells from an editor scope is a version-skew rejection and MUST NOT trigger blunt reset. Classify rejections as authority-failure (viewer writing CellsDoc, principal forgery) vs shape/version-skew (editor writing legacy-root cells), and only arm RESYNC_REQUIRED for the former. The taxonomy already distinguishes these; the reset trigger must respect the boundary.
  • Do not reject old-root cell writes during the data-move window at all. The `before.get(comm_id)` membership gate in `diff_comm_state` (runtime_agent.rs:1501) kept that orphan state from ever reaching the kernel, so it was silent, but it was permanent. When someone finally built the topology-absence GC pass the design's open questions had gestured at, it made things worse. The pass scanned RuntimeStateDoc for comm_ids with no topology entry and deleted the matching CommsDoc state. Underlying assumption. That deletion is a kind of read, so a document that is read-only to a principal can still be garbage-collected by that principal. The all-or-nothing model made the daemon a pure reader of CommsDoc state for the live-sync path, and the team never noticed that lifecycle reclamation is a *write* that the same principal must also own. Co-location of topology and state in one document had been silently providing that authority for free: `remove_comm` and `clear_comms` were single-document deletes, so whoever could delete topology automatically reclaimed state. The split severed the thing that made deletion authority follow topology authority, and no document boundary was drawn to put them back together.

Instead of a topology-absence scan that races writers, tombstone on close: the daemon's `remove_comm` writes a CommsDoc tombstone keyed by comm_id with the same change, so reclamation is causally tied to the close event rather than to an independently-replicated topology snapshot.   • Add a GC race regression test that opens a comm, writes state concurrently with a GC pass, and asserts the live state survives. If the only safe GC is tombstone-gated, this test forces that design and rejects the naive topology-absence scan.

The post-mortem found the arithmetic was always there: pre-split, a busy room paid 2 full-doc clones per write burst across 2 docs; post-split it paid up to 4 (Cells + Comms + RuntimeState + any future stream), and CommsDoc's write frequency was an order of magnitude higher than NotebookDoc's ever was. Underlying assumption. That auth-preview cost is a per-message constant worth ignoring, when it is actually O(doc-size) per write-frame and the split both multiplies the number of independently-cloned docs AND introduces CommsDoc, whose write frequency (per widget-interaction tick) is far higher than any pre-split doc. The design's recovery section even notes "make sync_message_new_changes just cheapen the preview by removing the doc/peer-state clone" as a future nicety, treating the clone as incidental rather than as the dominant cost that the doc-count multiplication makes load-bearing.

Today each frame independently re-clones from scratch.

That is the cruel irony of the north star here. Pre-split, the model had a per-change diff validator that at least inspected what was being written. Post-split, the team deleted all per-write inspection on the strength of "the doc boundary is the authorization boundary." But the doc boundary answers "what scope may write this doc," and it answers it against a scope value that was captured at connect and can never change for the life of the socket. The split made authorization coarser and more confident at the exact moment it removed the only thing that ran on every message. The contractor edited cells, repointed execution_ids, and triggered runs for another three days until their laptop went to sleep and the socket finally closed. There was no eviction path to close it sooner: grepping the cloud worker, the only `evict` is `evictStoredFrame` (notebook-room.ts:528), a frame-buffer GC with nothing to do with peers. The revoke handler at index.ts:967 never obtains the room's DO stub, never sends a SESSION_CONTROL close, never even tells the room a peer's authority changed. The Sever frame the recovery design nominated was still listed as "future work" (identity-and-trust.md:131), one of the costs that got deferred because it was net-new and the doc split was the headline. It was a three-day-stale snapshot. The blunt-reset recovery machinery, the ResyncRequired frame, the all-or-nothing reasoning — none of it engaged, because nothing was ever rejected. A revoked principal whose scope is frozen at "editor" produces zero rejections. The recovery system can only react to rejected writes, and revocation produces accepted ones.

Underlying assumption. That a connection's scope, captured once at connect and frozen onto the peer identity, remains a valid authorization fact for the entire socket lifetime — i.e. that ACL state and live-connection authority can never disagree. The whole all-or-nothing model reasons about "what scope may write this doc" but never about "is this scope still true right now." Authorization was treated as a connect-time gate, not a continuously-enforced session invariant.

On ACL revoke/downgrade in index.ts:967, after the D1 write, obtain the NotebookRoom DO stub for that notebook and call a new `notifyAclChanged(principal)` endpoint. Removing all per-message authorization inspection is only safe if the scope it trusts is guaranteed current. Add a CI/architecture test asserting that an ACL revoke produces a room-observable peer close, so the two halves can't ship decoupled.

The sever threshold had no principled value because every rejection was byte-identical, so operators chose between severing honest users and letting attackers churn clone-previews

The recovery frame shipped exactly as designed: every change-bearing rejection in `receive_notebook_sync` and `receive_runtime_state_sync` produces a generic `Err(JsError)` (crates/runtimed-wasm/src/lib.rs:595-597, 613, 618, 660-662, 678-684), the materializer's `receiveFrame` propagates it through `withHost` (apps/notebook-cloud/src/room-materializer.ts:65-79) with no taxonomy attached, and the room replies `ResyncRequired`. The escalation counter was bolted on as new per-connection room state (it had to be invented; there is none today in either the WASM host or notebook-room.ts, and it had to be persisted alongside the frozen identity across DO hibernation). Because the wire result carries no rejection KIND, the counter could only count "rejections-after-resync," exactly the undifferentiated signal the design admitted you cannot read intent from. Operators had one scalar, N, and three populations colliding on it: viewers fat-fingering a CellsDoc write, true-old clients still emitting legacy-root writes during the migration soak, and probes. The first prod tuning set N=3. Within a week the alert was honest flaky clients on bad mobile networks: a peer that resets, reconnects, and re-offers its still-unsynced local write before the snapshot fully lands trips the counter three times and gets a terminal SESSION_CONTROL sever, then reconnects into the same loop. Support tickets read "notebook keeps kicking me out." Ops raised N to 30 to stop the false severs. With N=30 the attacker problem became the outage. Each change-bearing frame, rejected or not, unconditionally clones the entire NotebookDoc AND the peer sync state, then runs a full `receive_sync_message_recovering` apply on the clone (lib.rs:600-613) purely to extract actor hashes and re-validate. There is no cheap pre-check that short-circuits a peer who was rejected on its last 30 frames; the expensive clone-and-apply happens first, every time, and only then does validation say no. A single connection holding a legitimate editor scope but deliberately re-offering a forbidden owner-field write 29 times per resync cycle forced 29 full-document clones per cycle inside a single-threaded WASM room host running in a Durable Object. On large notebooks the clone dominated DO CPU; the room stopped servicing honest peers' sync frames, checkpoints backed up, and the DO hit its CPU budget and reset. The reset re-bootstrapped every connected peer, which re-armed everyone's unsynced writes, which spiked clone load again. The feature got blamed for the outage in the postmortem because the sever counter was the named mechanism that "should have stopped the abuse" and didn't, because at N=30 it was tuned to never fire for the honest-client problem it had caused at N=3. The core trap: the design treated severity as a property you recover from observed repetition, but repetition is produced by network conditions and migration staleness just as readily as by malice, and the one signal that genuinely leans malicious, principal-forgery (`validate_room_notebook_change_actors`, lib.rs:864), was flattened into the same generic Err as a viewer's honest mistake. There was never a single N that separated the populations because they were never separable on the axis being measured.

Underlying assumption. That repetition-past-resync is a reliable proxy for malice. It is not: a flaky-network honest client and a stale migration-era client both re-offer the same rejected write after every reset for reasons that have nothing to do with intent, and the rejection bytes are identical to a probe's, so no single threshold N can separate the three populations on the one axis (count of rejections) the counter can actually measure.

Early warning signs

  • The sever/false-positive rate correlates with client network quality, not with any abuse signal: severed-peer events cluster on mobile/high-latency connections and on app versions still in the migration soak window, and the same principals reconnect and get severed again within seconds (a reset loop, not an attack).
  • Room-host CPU per frame and clone count climb with notebook size while accepted-write throughput stays flat: the DO is spending its budget on `receive_sync_message_recovering` clone-previews for frames it ultimately rejects, visible as a rising ratio of preview-applies to committed writes per connection before any sever fires.

Design changes

  • Attach a typed rejection reason to the wire result instead of a generic Err. The validators already know the kind at the rejection site (can_write gate vs validate_room_notebook_change_actors principal-forgery at lib.rs:864 vs editor-path vs runtime-state-scope at lib.rs:683). Propagate that enum through receive_peer_frame and into the ResyncRequired/Sever decision so principal-forgery (which an honest reset literally cannot produce) escalates on a different, far shorter counter than a viewer's honest wrong-doc write. Stop collapsing distinguishable failures into one undifferentiated count.
  • Make rejection cheap before making it punitive. Add an O(1) pre-validation gate that runs before the full doc+peer-state clone at lib.rs:605-606: the can_write scalar and the principal-binding check on raw change actor labels can reject without a full `receive_sync_message_recovering` apply. A peer with a known-bad last frame must not be able to force a full-document clone per offer. This removes the attacker-churn outage independent of any threshold choice.
  • Replace the count-based threshold with a token-bucket / rate-limit on rejected previews per connection that bounds CPU cost rather than guessing intent. The honest goal is not 'detect malice,' it is 'a connection cannot consume unbounded room CPU on rejected work.' Cap rejected-preview CPU per connection per window; an honest flaky client stays under the cap because it converges, an attacker hits the cap and is throttled (not severed), and the room stays responsive regardless of how N is tuned.
  • Gate the migration rejection class out of the counter entirely until the forward-tolerant-reader soak completes. A true-old client emitting legacy-root writes is staleness by construction and must be translated daemon-side (per the migration plan), never rejected-then-counted. Key the counter on a client-capability/version signal so migration-era clients cannot trip it at all.
  • Distinguish throttle from sever in the recovery frame design. Sever is terminal and reconnect re-arms the same loop, which is what turned false severs into support storms. Prefer a non-terminal backoff/throttle as the default response to repetition, reserve terminal sever for the one cryptographically-distinguishable case (principal-forgery after a clean resync), and make the client surface a hard build/seed-mismatch diagnostic for divergent-genesis instead of silently reset-looping into the counter.

The `widget_model_field(&entry.model_name, &entry.state, ...)` fallback (persist.rs:578) and `ensure_widget_state_model_field` on load (load.rs:408) only made sense when topology and state were one entry. Once split, reading identity out of the OTHER doc's state half hides exactly the desync this failure is about. Make topology the sole source of model identity and fail loudly if it is absent.
  • Add a round-trip reconstruction test that is the actual gate: open a notebook with N stateful widgets, mutate trait values, GC one comm via comm_close, save, reload, and assert the live `get_comms()` topology set, the CommsDoc state set, and the reloaded widget values all match the pre-save state exactly. This is the test that the current single-doc code passes trivially and the naive two-doc split fails - it must exist before the split lands.

Five frozen seeds and three seed-hash auth helpers outweigh deleting one validator

The team started Phase 1 in good faith: additive doc-id pointers (cells_doc_id, comms_doc_id) minted ensure_-style, the proven runtime_state_doc_id playbook. CommsDoc came first because it was the cheap new-doc case, and it shipped. Then the cost ledger of the actual north star came due, and it was front-loaded entirely onto permanent maintenance surface, not one-time work. To make CellsDoc and CommsDoc all-or-nothing per principal, each multi-principal doc needed its own is_canonical_*_seed_change hash-pin mirroring crates/notebook-doc/src/lib.rs:2400-2429, and each new frozen seed had to be registered in GENESIS_SEEDS_IN_WASM at crates/xtask/src/main.rs:1509-1518, which today lists exactly two tuples (notebook_genesis_v5.am, runtime_state_genesis_v2.am). Going to five docs meant five frozen .am assets, each one a #3086 divergent-genesis footgun if the gitignored wasm bundle drifts from source, plus three OnceLock-cached canonical_schema_seed_change_hashes helpers to keep in lockstep. Every one of those is a load-bearing invariant where a silent mismatch does not crash, it LWW-merges into a doc neither side reads. Worse, the payoff the effort was supposed to buy never materialized cleanly. Deleting validate_editor_notebook_changes (the cells-vs-root path diff at runtimed-wasm/src/lib.rs:917) was the headline win, but that validator is duplicated: the daemon path has its own validate_notebook_change_actors at notebook_sync_server/identity.rs:83 reached from peer_notebook_sync.rs:50, and the WASM room host has the parallel copy. So the split did not delete one validator, it forced both parallel implementations to learn the new five-doc shape identically or recovery behavior would diverge between daemon-local and cloud-room paths. On top of that, the genuinely hard residual problems the design itself enumerated, schema/version divergence in mixed-version rooms, scope revocation mid-session with no eviction path, divergent-genesis reset loops, all survived the split untouched because none of them were authorization-field problems. The team was now maintaining five seeds, three seed-hash helpers, dual validators, net-new per-doc reset machinery, a ResyncRequired/Sever control-frame protocol, a rejection-after-resync counter, and a multi-phase mixed-version migration, and the residual convergence bugs were still open. By December the calculus was obvious in retrospect: the partial-ownership wart was one within_cells path check on one diff, confined to the editor branch, with owners already bypassing it via can_write_all_notebook_changes. The ADR at docs/architecture/schema-evolution-and-genesis.md forbids in-place reshape of the frozen root, so cells and metadata could never be cleanly moved, only copied into new docs behind a Phase-0 forward-tolerant-reader soak that had to propagate through the entire fleet before the data-move could even begin. The team concluded that living with a single field-subtree path check was cheaper than permanently owning five genesis assets, three hash-pins, and a migration that risked silent data invisibility on every not-yet-updated client. The effort was shelved with CommsDoc shipped (a genuine win that stood on its own) and the NotebookDoc split abandoned.

Underlying assumption. That the north-star payoff is "delete one validator," when in reality the validator is duplicated across the daemon and WASM paths, gates only the editor branch, and is the single cheapest piece of the whole partial-ownership story, while the cost (five permanent frozen seeds, three seed-hash auth helpers, dual-validator parity, and an ADR-forbidden structural migration) is permanent maintenance surface that grows the divergent-genesis blast radius rather than shrinking it.

Early warning signs

  • The GENESIS_SEEDS_IN_WASM tuple list in crates/xtask/src/main.rs grows from 2 entries toward 5, and verify-genesis CI starts failing intermittently on PRs that touch any doc schema because a contributor rebuilt the daemon but not the gitignored wasm bundle (the exact #3086 drift). Frequency of these failures is the leading indicator the maintenance budget is being exceeded.
  • A PR review surfaces that a change to validate_editor_notebook_changes in runtimed-wasm/src/lib.rs was not mirrored into the daemon's validate_notebook_change_actors at notebook_sync_server/identity.rs:83, or vice versa, meaning the two enforcement copies have silently diverged. Once the docs go from 2 to 5, keeping these parallel validators in lockstep across every schema touch becomes a recurring reviewer burden.
  • The Phase-0 forward-tolerant-reader soak has no measurable end date because there is no fleet-version telemetry confirming old v5 clients have aged out, so the cells/metadata data-move keeps getting deferred sprint after sprint while CommsDoc (the only cheap piece) is the only thing that actually ships.

Design changes

  • Decouple the wins from the rebuild. Ship CommsDoc on its own merits (it collapses the RuntimeStateDoc carve-out and uses the cheap additive new-doc playbook) and explicitly do NOT couple it to the NotebookDoc owner/cells split. CommsDoc adds one seed and one hash-pin; the NotebookDoc split adds two more seeds, two more hash-pins, and the ADR-forbidden structural copy. Treat them as separate decisions so the cheap win is not held hostage to the expensive one.
  • Before committing to the split, retire the dual-validator liability first: unify the daemon (notebook_sync_server/identity.rs:83) and WASM (runtimed-wasm/src/lib.rs:917) validators into a single shared crate so 'delete one validator' is actually one deletion, not two parallel edits. If that consolidation is itself too costly, that is direct evidence the five-doc split's parity burden will be worse, and the data informs the go/no-go.
  • Add a fleet-version gate as a hard precondition in the plan: the cells/metadata data-move (the only part requiring the structural copy and the silent-data-invisibility risk) is forbidden until telemetry proves the forward-tolerant v6 reader has propagated past a threshold. Make CommsDoc and the additive pointers the only deliverables until that gate trips, so the expensive phase cannot start prematurely and stall mid-migration.
  • Re-scope the actual problem the validator solves. The within_cells path check exists only for the editor branch (owners already bypass it). Evaluate whether tightening that one diff (e.g. denying execution_id repoint to non-daemon-bound executions, the real residual open question) is sufficient for the authorization goals without splitting the doc at all, keeping the partial-ownership wart but neutering its sharp edge for a fraction of the maintenance cost.
  • Make the seed/hash maintenance cost visible and bounded: add a single source-of-truth table that maps each doc to its seed asset, its hash-pin helper, and its GENESIS_SEEDS_IN_WASM tuple, and a CI check that fails if a new frozen doc is added without all three. This does not reduce the count but converts a class of silent divergent-genesis failures into a loud build failure, lowering the per-doc carrying cost.

CommsDoc "anyone writes" dropped Guard A; orphan and forged widget state accumulated unbounded and was force-fed to the kernel

The split shipped pointer-first and looked clean. RuntimeStateDoc finally became all-or-nothing for clients, the comm-state allowlist vanished from receive_runtime_state_sync, and CommsDoc was stood up as a flat anyone-writes Automerge doc keyed by comm_id with a per-trait state map, modeled on the PoolDoc plumbing recipe. In the pre-split world, the only way to write comms/{comm_id}/state/{key} was set_comm_state_property / merge_comm_state_delta on RuntimeStateDoc, and both functions bail out with Ok(()) the moment the comm entry or its state map is absent (crates/runtime-doc/src/doc.rs:2502-2515 and 2536-2543). That early return is Guard A. It only works because topology and state were co-located: the daemon authored the comm entry on kernel comm_open before any client could land a state key, so a client writing state for a comm_id the runtime never opened simply hit the `let Some(... ) else { return Ok(()) }` and wrote nothing. When state moved to its own doc, there was no pre-existing entry to gate against. Any peer could splice a fresh comms/{forged_comm_id}/state/... subtree into CommsDoc, and because the doc is anyone-writes by design, the host had nothing to reject, the change merged, and it replicated to every peer. The orphan entries were now physically real and durable. The second half is what made it dangerous rather than merely messy. The team believed the runtime before-membership gate had been "re-homed" into the CommsDoc receive path, but the re-home was not faithful. The surviving safety was diff_comm_state (crates/runtimed/src/runtime_agent.rs:1495-1521), which only forwards a comm to the kernel when before.get(comm_id) is Some — and "before" is a snapshot of the daemon's own RuntimeStateDoc comms (read_state().comms, doc.rs:3187), the topology doc. In the new world the comm-state diff was computed against CommsDoc, whose membership is whatever any peer wrote, while the kernel-forward gate's `before` snapshot was rewired to CommsDoc membership too (so "is this comm known?" now answered "yes, a peer just wrote it") instead of staying anchored to RuntimeStateDoc topology. With the membership check answering against attacker-controlled CommsDoc keys, an orphan comm_id passed the before.get(comm_id).is_some() test, produced a non-empty delta, and diff_comm_state happily emitted it. The runtime agent's single kernel-forward consumer (runtime_agent.rs:348-368) then sent comm_msg traffic for comms the kernel never opened — synthetic widget state delivered straight into the kernel process, plus an ever-growing CommsDoc that every reconnecting peer had to download and merge. By the time anyone noticed, CommsDoc snapshots for long-lived hosted rooms were tens of megabytes of dead comm_ids that no widget referenced, reconnection got slow because the snapshot ballooned, and a security review flagged that a viewer-scope peer could inject arbitrary comm_msg payloads to the kernel by writing orphan CommsDoc state — the exact "force-fed to the kernel" outcome. The all-or-nothing authorization win was real, but it had quietly converted a co-location safety invariant into a missing runtime check, and nobody had a test that exercised "write state for a comm the runtime never opened" because in the old model that path was structurally impossible.

Underlying assumption. That Guard A (the orphan-state no-op) was an authorization concern that the document split subsumed, when it was actually a topology-membership invariant enforced incidentally by topology and state being in the same document. Once they live in separate docs, "the comm exists" must be re-derived from RuntimeStateDoc topology on every CommsDoc receive; it cannot be inferred from CommsDoc's own keys, because CommsDoc membership is exactly the untrusted thing anyone can write.

Early warning signs

  • CommsDoc snapshot size for a hosted room grows monotonically and never shrinks on widget close, while the RuntimeStateDoc comms (topology) map stays small — the gap between CommsDoc comm_id count and RuntimeStateDoc topology comm_id count is the orphan count and it should be zero. A simple metric of len(CommsDoc.comms) minus len(RuntimeStateDoc.comms) trending above zero is the canary.
  • Kernel receives comm_msg for a comm_id with no preceding comm_open in the same session, or [comm_*] debug logs show forwarded deltas for comm_ids absent from the daemon's pre-sync topology snapshot. The pre-split diff_comm_state log line 'Skipping comm forward' (runtime_agent.rs:361) stops firing for cases it used to catch, because the before-snapshot now contains attacker-written ids.

Design changes

  • Anchor the kernel-forward membership gate to RuntimeStateDoc topology, never to CommsDoc keys. diff_comm_state's `before` snapshot (runtime_agent.rs:1495-1521, fed from read_state().comms) must keep reading the daemon-owned RuntimeStateDoc comm entries, and the gate condition must be 'comm_id exists in RuntimeStateDoc topology', explicitly independent of whether CommsDoc has a state subtree for it. Make this an intersection: forward state only for comm_ids present in BOTH docs.
  • Re-create Guard A as an explicit runtime-side drop on CommsDoc receive: when a CommsDoc change introduces a comms/{comm_id}/state subtree whose comm_id has no RuntimeStateDoc topology entry, the daemon ignores it for kernel-forward purposes (and ideally GC-sweeps it). Add the exact test that the old model made impossible — write CommsDoc state for a comm_id the runtime never opened, assert zero kernel forwards and that the orphan is not propagated — mirroring test_set_comm_state_property_nonexistent (doc.rs:5691) but at the CommsDoc boundary.
  • Add a topology-absence GC pass owned by the daemon: on comm_close the runtime already deletes the whole entry via remove_comm; give the daemon CommsDoc write authority (or a reaper keyed on RuntimeStateDoc topology) so closed/orphan comm state is reclaimed and CommsDoc cannot grow unbounded. Without an owner for deletion, even legitimate closed-comm state leaks.
  • Emit the orphan-count metric (CommsDoc comm_ids minus RuntimeStateDoc topology comm_ids) and the 'forwarded comm_msg without prior comm_open' counter as first-class observability, and gate the data-move rollout on both reading zero in soak. Treat any non-zero as a release blocker, since the failure is silent accumulation, not a crash.
  • Keep the per-field foreign-actor echo-suppression filter (runtime_agent.rs:348-352, the rt:kernel: actor check) as a distinct, explicitly-documented concern inside the CommsDoc receive path. It is anti-amplification, not authorization, and re-homing it is necessary but does NOT substitute for the topology-membership gate; conflating the two is how the gate got dropped.

execution_id repointing surfaces another principal's outputs because the daemon-bound cell binding was never enforced

The five-doc split shipped on schedule. Authorization became clean and document-shaped: editors write CellsDoc, the daemon owns RuntimeStateDoc, and `validate_editor_notebook_changes` was deleted because cells and owner-metadata no longer share a document. The team correctly reasoned that editors can no longer write output bytes: outputs live in RuntimeStateDoc (executions/{id}/outputs/{oid}), which editors have zero write access to. The one residual the design wrote down explicitly in `outputAuthority` and `openQuestions` was "validate that a cell execution_id belongs to a daemon-bound execution to prevent surfacing another principal's outputs; do not move the pointer to RuntimeStateDoc since editors must also clear it." That sentence was the whole ballgame, and it never became code. Across the migration churn it read like a nice-to-have hardening item, not a gate, so it sat in the backlog while the split landed. The actual exploit is small. An editor's only legal output-adjacent write is the `execution_id` pointer under their own cell, allowed by the cells allowlist (the doc comment at runtimed-wasm/src/lib.rs:907-913 says editors "may write" execution_count/execution_id and asserts "this cannot fabricate live execution state" because RuntimeStateDoc is gated separately). True, they can't fabricate state. But they don't need to. `NotebookHandle::set_execution_id` (notebook-doc/src/lib.rs:1892) just puts an arbitrary string into `cells/{id}/execution_id`. A malicious editor enumerates or guesses an execution_id from a different principal's run, or from a cell whose outputs the room ACL never granted them, and points their own cell at it. There is no validation anywhere in the editor write path that the target execution exists, that it was queued for this cell, or that the room granted this principal visibility into it. The pointer is a free-form scalar. The frontend then surfaces it with zero suspicion, because output resolution is pure pointer-following. `getCellOutputsSnapshot` (notebook-outputs.ts:286-298) reads `getCellExecutionId(cell_id)`, looks up `getExecutionById(executionId)`, and renders whatever output_ids it finds. `materialize-cells.ts:28` does the same: `getRuntimeState().executions[executionId]` with no comparison against the cell. The bitter irony is that the daemon already records the correct binding: the execution entry in RuntimeStateDoc carries `cell_id` (runtime-doc/src/doc.rs:201, written at doc.rs:1271-1272). The frontend has the field it would need to reject the mismatch - `execution.cell_id !== cell_id` - sitting right there in the resolved snapshot, and never reads it. So a repointed cell renders foreign outputs (a different user's dataframe, an error traceback leaking a file path or secret, a plot built from data the editor was never authorized to see) as if they were its own. RuntimeStateDoc stayed perfectly read-only and "all-or-nothing"; the leak rode entirely on a pointer the editor was always allowed to write, dereferenced by a client that trusts the pointer absolutely.

Underlying assumption. That "editors cannot write outputs" is equivalent to "editors cannot surface outputs they aren't authorized to see." The design proved the first (RuntimeStateDoc is daemon-only, editors can't mint output bytes) and treated the second as following from it. It doesn't: read-access to outputs is mediated entirely by an editor-writable pointer (cells/{id}/execution_id) that the frontend dereferences with no check that the pointed-at execution was actually bound to that cell by the daemon - even though the binding (execution.cell_id) exists in the data.

Early warning signs

  • A cell renders an execution_count or outputs that don't match its own last run - e.g. In[7] outputs appearing on a cell that was never executed in this session, or a traceback referencing a file/variable that doesn't exist in this cell's source. This is the user-visible tell that a pointer points somewhere it shouldn't.
  • grep for the resolution sites (apps/notebook/src/lib/notebook-outputs.ts:288, materialize-cells.ts:28) shows they read getRuntimeState().executions[executionId] but never compare execution.cell_id against the cell doing the rendering - the binding field is present in the snapshot and unread.
  • The open question 'can a malicious editor repoint a cell execution_id at another principal execution to surface foreign outputs' is still sitting in the design doc's openQuestions list with no corresponding test (no test named like repoint_foreign_execution_rejected) and no code path that calls execution.cell_id during resolution. A security residual that survives to merge with no test is the warning sign.

Design changes

  • Enforce the cell-binding at resolution time on the client: in getCellOutputsSnapshot/useCellOutputs/materialize-cells, after resolving executionId -> execution, drop the outputs if execution.cell_id is set and !== cell_id. The field already exists (runtime-doc/src/doc.rs:201); this is a cheap guard at the two read sites and fails closed (render nothing) on mismatch.
  • Make the daemon the sole writer of execution_id and demote the editor's pointer. Editors legitimately need to CLEAR (set null, for clear-outputs) but not to SET an arbitrary execution_id - the daemon stamps it at queue time (execute_cell.rs:308, run_all_cells.rs:246). Tighten the cells allowlist so editors may only null execution_id, and reject any editor write that sets execution_id to a non-null value the daemon didn't author. This closes the forgery at the write boundary instead of trusting the read boundary.
  • Add a room-host validation in receive_notebook_sync: when an editor change sets cells/{id}/execution_id to a non-null value, verify the target execution exists in RuntimeStateDoc AND its cell_id matches the cell being pointed (cross-doc check the daemon can do, the cloud host may need the RuntimeStateDoc snapshot). Reject otherwise. This makes the repoint a wrong-doc-shaped rejection that the recovery design already handles (RESYNC_REQUIRED).
  • Write the regression test the open question implies: editor repoints cell-A.execution_id at cell-B's (or another principal's) execution_id; assert the outputs do NOT render for cell-A. Gate the five-doc rollout on this test passing, so the residual cannot ship as a backlog item.
  • Treat execution_id as principal-scoped at mint time: include the authoring principal in the execution record and have the client refuse to surface an execution whose principal the current room ACL does not grant output-read access to. Defends even against a same-cell-id collision across principals.
